# 第三方员工同步接口文档

更新时间：2026-06-20

## 1. 文档目的

本文档说明怀仁商学院系统当前“用户信息从第三方 HR/员工通讯录同步”的接口、配置、签名规则、返回结构和本地落库逻辑。

当前实现以第三方员工接口为主数据源：

- 管理员在后台发起员工同步预览。
- 系统服务端签名请求第三方 `/api/open/employees`。
- 管理员确认后执行同步，更新或创建本地用户。
- 同步结果写入同步批次和明细表，便于追踪。

涉及代码：

- `backend/app/employee_open_client.py`
- `backend/app/employee_sync.py`
- `backend/app/users_api.py`
- `backend/app/models.py`

## 2. 角色与权限

| 角色 | 权限 |
| --- | --- |
| 管理员 / 超级管理员 | 可查询第三方员工、预览同步、执行同步、查看同步日志、删除同步日志 |
| 普通用户 | 无同步接口访问权限 |
| 第三方 HR 系统 | 提供员工列表接口，由本系统服务端签名调用 |

所有本系统后台同步接口均要求管理员登录态：

```http
Authorization: Bearer <admin_token>
```

## 3. 服务端配置

配置文件：`backend/.env`

> 注意：密钥配置项必须放入环境配置文件；文档只列配置项名和占位值，不建议在文档、聊天或代码仓库里明文暴露真实私钥。

```env
# 第三方员工通讯录同步开关
EMPLOYEE_SYNC_ENABLED=true

# 第三方 HR OpenAPI 基础地址，不带最后的接口路径
EMPLOYEE_SYNC_BASE_URL=https://<third-party-hr-host>

# 第三方分配给本系统的应用 ID
EMPLOYEE_SYNC_APP_ID=<app_id>

# 本系统用于请求签名的 RSA 私钥，支持 PEM 文本或 Base64 DER
EMPLOYEE_SYNC_PARTNER_PRIVATE_KEY=<partner_private_key>

# 第三方 HR 用于响应验签的 RSA 公钥，支持 PEM 文本或 Base64 DER
# 可选；未配置时不做响应签名校验
EMPLOYEE_SYNC_HR_PUBLIC_KEY=<hr_public_key>

# 请求第三方接口超时时间，单位秒；实际最小值为 5 秒
EMPLOYEE_SYNC_TIMEOUT_SECONDS=15

# 第三方有员工但本地没有账号时，是否自动创建本地用户
EMPLOYEE_SYNC_AUTO_CREATE_USER=true

# 第三方缺失本地用户时，非初始化模式下是否自动置离职并禁用
WECOM_SYNC_DISABLED_USERS=true

# 同步保护状态：本地账号处于这些状态时，即使第三方缺失，也不会自动置离职
WECOM_SYNC_PROTECTED_STATUSES=试岗,离职
```

第三方同步启用条件：

- `EMPLOYEE_SYNC_ENABLED=true`
- `EMPLOYEE_SYNC_BASE_URL` 非空
- `EMPLOYEE_SYNC_APP_ID` 非空
- `EMPLOYEE_SYNC_PARTNER_PRIVATE_KEY` 非空

当前本地 `backend/.env` 检查结果：以上第三方同步相关配置均已填写；密钥值已脱敏，不在本文档展示明文。

## 4. 第三方员工接口

### 4.1 接口地址

```http
GET {EMPLOYEE_SYNC_BASE_URL}/api/open/employees
```

本系统代码固定调用路径：

```text
/api/open/employees
```

### 4.2 查询参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `name` | string | 否 | 员工姓名，后台搜索第三方员工时使用 |
| `mobile` | string | 否 | 手机号，后台搜索第三方员工时使用 |
| `external_user_id` | string / int | 否 | 第三方员工 ID，后台搜索第三方员工时使用 |

同步预览和执行时不传筛选参数，会拉取第三方返回的完整员工列表。

### 4.3 请求头

| Header | 必填 | 说明 |
| --- | --- | --- |
| `X-App-Id` | 是 | `EMPLOYEE_SYNC_APP_ID` |
| `X-Timestamp` | 是 | Unix 秒级时间戳 |
| `X-Sign` | 是 | RSA-SHA256 签名后 Base64 字符串 |

### 4.4 签名规则

签名算法：

```text
RSA-SHA256 with PKCS#1 v1.5 padding
```

签名原文：

```text
{app_id}
{timestamp}
{method}
{path}
{canonical_query}
```

本接口请求示例中的签名原文：

```text
shangxue-app
1781920800
GET
/api/open/employees
mobile=13800138000&name=张三
```

`canonical_query` 规则：

- 仅包含值不为 `None` 的参数。
- 按参数名升序排序。
- 拼接格式为 `key=value`。
- 多个参数用 `&` 连接。
- 无参数时为空字符串。

示例：

```text
external_user_id=123&mobile=13800138000&name=张三
```

### 4.5 第三方响应格式

成功响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [
      {
        "external_user_id": 123,
        "name": "张三",
        "mobile": "13800138000",
        "department_name": "销售部",
        "position": "销售经理",
        "rank_name": "M3",
        "status": 1,
        "employment_status": 1,
        "wecom_userid": "zhangsan"
      }
    ]
  }
}
```

失败响应：

```json
{
  "code": 40001,
  "message": "签名错误",
  "data": {}
}
```

响应说明：

- `code=0` 代表成功。
- `data.list` 必须是数组。
- 数组元素必须是对象；非对象元素会被忽略。
- `code != 0` 时，本系统会把 `message` 作为错误信息返回给后台。

### 4.6 第三方响应验签

第三方可在响应头返回：

```http
X-Response-Sign: <base64_signature>
```

本系统验签规则：

- 使用 `EMPLOYEE_SYNC_HR_PUBLIC_KEY` 加载第三方 HR 公钥。
- 对完整响应 body 字符串做 RSA-SHA256 验签。
- 验签失败则同步接口返回 400。

兼容逻辑：

- 未配置 `EMPLOYEE_SYNC_HR_PUBLIC_KEY` 时跳过响应验签。
- 已配置公钥但响应头没有 `X-Response-Sign` 时，当前代码也会跳过响应验签。

## 5. 第三方字段映射

| 第三方字段 | 类型 | 本地字段 / 用途 |
| --- | --- | --- |
| `external_user_id` | int/string | 同步预览展示，不落用户主表 |
| `name` | string | `users.display_name`、`users.real_name` |
| `mobile` | string | `users.username`；新建或手机号变更时，密码重置为手机号后 6 位的 MD5 |
| `department_name` | string | `users.department` |
| `position` | string | `users.position` |
| `rank_name` | string | `users.rank_name`；同时映射 `users.job_level` |
| `status` | int | 本地任职状态：`1 -> 转正`，`2 -> 试用` |
| `employment_status` | int | 本地账号启停：`1 -> active/未禁用`，`2/3 -> inactive/禁用` |
| `wecom_userid` | string | `users.wecom_userid` |
| 完整员工对象 | object | `users.wecom_raw_json` |

`rank_name` 到 `job_level` 的映射：

| `rank_name` 前缀 | `job_level` |
| --- | --- |
| `M` | `M线` |
| `P` | `P线` |
| `L` | `L线` |
| 其他 / 空 | 不覆盖本地已有值 |

`status` 映射：

| 第三方 `status` | 本地 `employment_status` |
| --- | --- |
| `1` | `转正` |
| `2` | `试用` |
| 其他 / 空 | 不覆盖本地已有值 |

`employment_status` 映射：

| 第三方 `employment_status` | 本地 `status` | 本地 `disabled` |
| --- | --- | --- |
| `1` | `active` | `false` |
| `2` | `inactive` | `true` |
| `3` | `inactive` | `true` |
| 空 | `active` | `false` |

### 6.3 执行员工同步

```http
POST /api/admin/users/employee-sync/execute
```

请求体：

```json
{
  "initial_mode": true,
  "preview_token": "Q5s9w2Xb...example"
}
```

请求参数说明：

| 参数 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `initial_mode` | boolean | 否 | `true` | 是否初始化模式 |
| `preview_token` | string/null | 否 | `null` | 预览接口返回的令牌 |

执行逻辑：

- `preview_token` 有效时，按该预览结果执行。
- `preview_token` 为空、过期或已被消费时，后端会重新请求第三方并生成预览后执行。
- 如果传入的 `initial_mode` 与缓存预览里的 `initial_mode` 不一致，后端会重新生成预览。

返回示例：

```json
{
  "batch_id": 1001,
  "initial_mode": true,
  "summary": {
    "matched": 30,
    "bound": 20,
    "updated": 30,
    "created": 3,
    "left": 0,
    "disabled": 0,
    "conflict": 1,
    "skipped": 2
  }
}
```

返回字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `batch_id` | int | 同步批次 ID |
| `initial_mode` | boolean | 实际执行模式 |
| `summary` | object | 执行结果统计 |

`summary` 字段说明：

| 字段 | 说明 |
| --- | --- |
| `matched` | 匹配到本地用户并应用更新数量 |
| `bound` | 通过手机号绑定到本地用户数量 |
| `updated` | 更新本地用户数量 |
| `created` | 自动新建用户数量 |
| `left` | 置离职数量 |
| `disabled` | 禁用账号数量 |
| `conflict` | 冲突数量 |
| `skipped` | 跳过数量 |

## 7. 同步动作与状态

### 7.1 预览动作 action

| action | 触发条件 | 执行结果 |
| --- | --- | --- |
| `update_bound` | 第三方 `wecom_userid` 已绑定本地用户 | 更新本地用户资料；手机号变化时更新账号并重置密码 |
| `bind_by_mobile` | 手机号匹配本地唯一用户，姓名也匹配或可唯一确认 | 绑定 `wecom_userid` 并更新资料 |
| `update_by_name` | 姓名唯一匹配，但手机号不同 | 更新本地账号手机号，密码重置为新手机号后 6 位 |
| `pending_create` | 第三方有员工，本地没有对应用户 | 若允许自动创建，则新建本地用户 |
| `mark_left` | 非初始化模式下，本地用户在第三方缺失且未受保护 | 置为离职并禁用 |
| `local_unbound` | 本地用户未匹配第三方员工 | 只记录或提示，不更新用户 |
| `conflict` | 手机号重复、姓名冲突、绑定冲突等 | 跳过，需要人工处理 |
| `skip_missing_identity` | 第三方员工同时缺少手机号和 `wecom_userid` | 跳过 |

### 7.2 执行明细 status

| status | 说明 |
| --- | --- |
| `applied` | 已应用到本地用户 |
| `created` | 已创建本地用户 |
| `skipped` | 已跳过 |

### 7.3 匹配方式 match_type

| match_type | 说明 |
| --- | --- |
| `wecom_userid` | 通过第三方 `wecom_userid` 匹配 |
| `mobile` | 通过手机号匹配 |
| `mobile_name` | 手机号重复时进一步通过姓名唯一确认 |
| `name` | 通过姓名唯一匹配 |
| 空字符串 | 无本地匹配或无需匹配 |

| `real_name` | 第三方姓名 |

## 11. 定时同步现状

项目中存在定时同步 worker：

- `backend/app/employee_sync_worker.py`

该 worker 设计为每天 08:00 执行一次：

```text
mode=scheduled
initial_mode=false
```

但当前 `backend/app/main.py` 启动钩子没有挂载 `employee_sync_worker`，因此当前实际可用入口主要是管理员手动预览、手动执行和查询同步记录。

如需启用定时同步，需要在 `main.py` startup/shutdown 中增加 worker task 管理，并确认生产环境只运行一个定时实例，避免多进程重复同步。

## 12. 第三方对接检查清单

第三方 HR 系统需要确认：

- 提供 `GET /api/open/employees`。
- 支持 `name`、`mobile`、`external_user_id` 筛选。
- 校验请求头 `X-App-Id`、`X-Timestamp`、`X-Sign`。
- 使用同样的签名原文格式进行验签。
- 成功返回 `code=0`。
- 员工数组放在 `data.list`。
- 员工对象至少建议包含 `name`、`mobile`、`department_name`、`position`、`rank_name`、`status`、`employment_status`、`wecom_userid`。
- 如需响应验签，返回 `X-Response-Sign`，签名内容为完整响应 body 字符串。

本系统部署前需要确认：

- `EMPLOYEE_SYNC_ENABLED=true`。
- `EMPLOYEE_SYNC_BASE_URL` 指向正确环境。
- `EMPLOYEE_SYNC_APP_ID` 与第三方一致。
- `EMPLOYEE_SYNC_PARTNER_PRIVATE_KEY` 与第三方验签公钥配套。
- `EMPLOYEE_SYNC_HR_PUBLIC_KEY` 与第三方响应签名私钥配套。
- 先使用“同步预览”检查 `conflict` 数量。
- 第一次同步建议 `initial_mode=true`，避免误把历史本地用户置离职。
- 日常同步或定时同步可使用 `initial_mode=false`。
