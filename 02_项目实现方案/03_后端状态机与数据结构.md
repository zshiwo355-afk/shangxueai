# 后端状态机与数据结构

> 本文档聚焦销售训练 / 场景通关的会话状态机；其它业务（试卷、魔学院、推送）的状态机散落在各 `*_api.py`，由数据库字段直接驱动，按需查阅 `backend/app/models.py`。

## 一、训练 / 通关 Session 数据结构（[backend/app/schemas.py](backend/app/schemas.py)）

```python
class SessionState(BaseModel):
    session_id: str
    created_at: datetime
    training_type: str
    difficulty: str
    customer_type: str
    visible_brief: VisibleBrief                  # 前端可见
    hidden_training_pack: dict[str, Any]         # 永远不下发前端
    round_count: int = 0
    current_stage: str = "opening"
    chat_history: list[ChatTurn]
    score_trace: list[ScoreTrace]
    emotion_state: EmotionState
    completed_actions: CompletedActions
    is_finished: bool = False
    first_customer_message: str = ""
```

JSON 形态（与 V1 保持一致）：


落库表：`training_sessions`（实时会话）、`training_records`（结束后的复盘 + 全文）、`exam_attempts`（场景通关）。

## 二、Chat History

`ChatTurn` 列表，按轮次顺序追加：

```json
[
  {"round": 1, "role": "customer", "content": "你好，我想了解一下你们这个酒。", "stage": "opening"},
  {"round": 1, "role": "trainee", "content": "王总您好，您这边是自饮还是送礼？", "stage": "need_probe"}
]
```

`stage` 由后端在每轮处理后写入，不依赖前端传入。

## 三、Score Trace

```json
[
  {
    "round": 1,
    "round_score": 8,
    "hit_points": [],
    "missed_points": [],
    "risk_points": [],
    "emotion_delta": {},
    "completed_actions_delta": {},
    "next_customer_strategy": ""
  }
]
```

`emotion_delta` / `completed_actions_delta` 仅记录本轮变化，便于做趋势图与复盘溯源。

## 四、状态机

阶段（按 SOP 排序）：

- `opening`
- `need_probe`
- `brand_trust`
- `product_intro`
- `price_discuss`
- `objection`
- `closing`
- `after_sale`
- `finished`

推进规则：

- LLM 在每轮评分输出中给出 `suggested_next_stage`。
- 后端 `state_machine.build_state_view` + `chat_pipeline` 校验后再决定是否前进，禁止跨阶段跳跃。
- `current_stage = finished` 仅由 `/finish` 接口或异常分支设置。

## 五、可见 / 不可见字段

- 可见（`StateView` + `VisibleBrief`）：`round_count`、`min_rounds`、`current_stage`、`emotion_label`、`can_finish`、`training_title`、`training_type`、`difficulty`、`exam_scope`、`trainee_notice`。
- 不可见（仅服务端持有）：`hidden_training_pack`（完整客户画像、预算、底牌、成交阈值、隐藏异议池）、`emotion_state` 原始数值、`score_trace` 完整内容。

## 六、成交硬判断

后端 `chat_pipeline` 在 `/finish` 阶段叠加工程级硬规则（与知识库评分独立）：

```python
def can_deal(session, settings):
    e = session.emotion_state
    a = session.completed_actions
    return (
        session.round_count >= settings.min_rounds
        and e.trust >= 70
        and e.interest >= 65
        and e.deal_willingness >= 65
        and e.price_resistance <= 55
        and a.recommended_product
        and a.handled_objection
        and a.attempted_close
    )
```

任一不满足即强制 `result="未成交"`，`deal_reason` 由 LLM 输出，但 `result` 与 `is_pass` 由后端最终决定。

## 七、复盘落库

训练结束后：

- 写 `training_records`（一行 = 一次完整复盘），包含原始 `chat_history` / `score_trace` / 复盘 JSON。
- 用户端 `/api/training/records`、管理端 `/api/admin/training-records` 都读这张表。
- 通关链路同时更新 `exam_attempts.status` 为 `completed` / `pending_review` / `reviewed`。

## 八、其它业务的关键状态字段

| 业务 | 表 | 状态字段 | 主要枚举值 |
| --- | --- | --- | --- |
| 试卷 | `papers` | `status` | `draft` / `published` / `archived` |
| 试卷派发 | `paper_assignments` | `status` / `wecom_push_status` | `pending` / `in_progress` / `submitted` / `graded` ; `none` / `pending` / `sent` / `failed` |
| 试卷提交 | `paper_submissions` | `status` | `in_progress` / `submitted` / `graded` |
| 试卷题答 | `paper_answers` | `is_correct` / `auto_score` / `ai_score` / `manual_score` / `final_score` | 数字或布尔 |
| 通关 | `exams` | `status` | `pending` / `in_progress` / `completed` / `pending_review` / `reviewed` |
| 通关尝试 | `exam_attempts` | `status` | 同上 + `failed` |
| 推送 | `notification_logs` | `status` | `pending` / `sent` / `failed` |
| 魔学院进度 | `magic_video_progress` | `status` | `not_started` / `in_progress` / `completed` |
| 员工同步 | `wecom_sync_entries` | `action` | `update` / `bind` / `mark_left` / `disable` / `conflict` / `skip` |
