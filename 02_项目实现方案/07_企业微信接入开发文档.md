# 企业微信接入开发文档

> 本文档用于后续正式接入企业微信时作为开发依据。当前阶段不改业务代码，只整理现有系统需要预留和改造的点。

## 1. 接入目标

企业微信接入后，系统需要支持以下能力：

- 企业微信应用内免密码登录，用户从企业微信工作台进入后直接进入系统。
- 用户信息可从企业微信通讯录同步，用来维护本地用户表快照。
- 考试派发、提交、评分/复核结果、待办提醒可以通过企业微信应用消息触达用户或管理员。
- 后续多个系统都接企业微信时，优先复用同一套登录/身份方案，避免每个系统都重复登录。

## 2. 当前项目现状

### 2.1 认证登录

当前登录体系在 `backend/app/auth.py`：

- `/api/auth/login`：用户名 + 密码登录。
- `/api/auth/me`：通过 JWT 获取当前用户。
- `/api/auth/logout`：前端清除 token。
- `create_token(user_id, username, role)`：签发现有 JWT。
- 前端 `frontend/src/lib/auth.js` 将 token 和 currentUser 存在 `localStorage`。
- `frontend/src/App.jsx` 通过 `RequireAuth` / `RequireAdmin` 控制路由访问。

重要结论：

- 当前 JWT 体系可以保留。
- 企业微信登录成功后，不需要重做全套鉴权，只需要在后端通过企业微信身份找到本地用户，然后复用 `create_token` 签发系统自己的 JWT。

### 2.2 用户表

当前用户模型在 `backend/app/models.py` 的 `User`：

- `username`：当前本地实际使用手机号作为用户名。
- `password_md5`：本地密码。
- `display_name`
- `real_name`
- `department`
- `position`
- `role`：`user` / `admin` / `super_admin`
- `is_newcomer`
- `employment_status`
- `status`
- `disabled`

重要结论：

- 因为本地 `username` 已经是手机号，企微登录时可以用“企业微信手机号”匹配 `users.username`。
- 本地用户表本质上仍然是业务快照，不建议完全依赖实时企微查询，否则考试、派发、历史记录会被外部通讯录变动影响。

### 2.3 试卷考试链路

当前更完整的“试卷考试”链路在 `backend/app/papers_api.py`、`backend/app/paper_assignments_api.py`：

- 管理员建题库、建试卷、发布试卷。
- 管理员通过 `paper_assignments` 给用户派发试卷。
- 用户在 `/papers` 查看待考试卷。
- 用户进入 `/papers/:assignmentId/take` 作答。
- 用户提交后生成 `paper_submissions` 和 `paper_answers`。
- 客观题自动判分。
- 简答题可走 AI 评分；无法完成或需要人工复核时保持 `submitted`。
- 管理员在待复核列表评分，最终进入 `graded`。

现有企微预留：

- `backend/app/wecom_push.py` 已有 `push_assignment(assignment_id, db)` stub。
- `paper_assignments` 已有 `wecom_push_status`、`wecom_push_payload_json`、`wecom_push_error`、`wecom_pushed_at` 字段。
- 管理端已有单个推送 `/api/admin/paper-assignments/{assignment_id}/wecom-push`。
- 管理端已有批量推送 `/api/admin/paper-assignments/bulk-wecom-push`。

重要结论：

- 试卷派发推送已有接口和字段基础，后续应优先替换 `wecom_push.py` stub，而不是另起一套派发通知逻辑。

### 2.4 场景通关链路

当前“场景通关”链路在 `backend/app/exams_api.py`：

- 管理员创建或批量创建 `exams`。
- 用户通过 `/exam/:examId/intro` 开始通关。
- 开始后创建 `exam_attempts`，并进入销售对练/通关会话。
- 用户完成后调用 LLM 生成复盘结果，`exam_attempts.status = completed`。
- 当前逻辑仍保留管理员复核入口，`exam.status = pending_review`。

重要结论：

- 如果后续仍保留该链路，需要补齐“通关派发通知”“通关完成通知”“通关结果通知”。
- 如果业务重心转向试卷派发链路，需要明确是否继续维护 `exams/exam_attempts`。

### 2.5 魔学院学习链路

当前魔学院相关代码在 `backend/app/magic_academy_api/*` 和 `frontend/src/components/MagicAcademyPage.jsx`：

- 视频课程、视频系列、读书内容、读书系列、音频打卡、题点测验。
- 内容可按全部员工、部门、岗位、在职状态、指定员工派发或展示。
- 当前主要依赖前端列表、统计、补卡设置，没有正式企业微信消息触达。

重要结论：

- 后续企业微信通知不只考试要接，魔学院也应接入学习提醒、打卡提醒、补卡提醒、未完成提醒。

## 3. 企业微信登录方案

### 3.1 推荐登录流程

1. 用户从企业微信工作台打开系统应用。
2. 前端发现没有本系统 token。
3. 前端跳转后端 `/api/auth/wecom/start?redirect=/home`。
4. 后端生成企业微信 OAuth 授权地址，并携带 `state`。
5. 企业微信回调 `/api/auth/wecom/callback?code=xxx&state=xxx`。
6. 后端用 `code` 换取企业微信访问用户身份，拿到企微 `userid`。
7. 后端用 `userid` 查询通讯录成员详情，拿到 `mobile`。
8. 后端规范化手机号，用 `users.username == mobile` 查本地用户。
9. 如果用户存在且未禁用，调用现有 `create_token` 签发 JWT。
10. 后端跳回前端回调页，前端写入 token/currentUser，再进入原目标页面。

### 3.2 手机号匹配规则

本项目当前最适合的主匹配规则：

- 企业微信通讯录手机号 `mobile` 对应本地 `users.username`。

必须统一：

- 本地 `username` 必须全员使用手机号。
- 手机号统一不带国家码，例如 `13800000000`。
- 如果企微返回 `+86 13800000000` 或 `8613800000000`，后端需要规范化后再匹配。
- 如果匹配不到本地用户，不自动创建用户，默认提示“账号未开通，请联系管理员”。

### 3.3 是否需要保存企微 userid

短期最小改造：

- 不新增字段，仅使用企业微信手机号匹配本地 `users.username`。

推荐改造：

- 在 `users` 增加 `wecom_userid`，用于后续消息推送、身份稳定性和审计。
- 首次企微登录或用户同步时，根据手机号匹配成功后写入 `wecom_userid`。

原因：

- 企业微信应用消息通常面向 `userid` 发送。
- 手机号可能变更，`userid` 更稳定。
- 后续多个系统接入时，`wecom_userid` 方便做统一身份。

折中原则：

- 业务主关联仍以手机号匹配本地用户。
- 技术字段 `wecom_userid` 仅作为企微接口调用和缓存，不替代本地用户 ID。

## 4. 用户信息同步方案

### 4.1 同步原则

企业微信接入后，用户信息同步不是单纯导入字段，而是要让企业微信成为账号在职/离职/禁用状态的主要来源，同时保留本地提前建新人账号的能力。

核心原则：

- 本地 `users` 仍然是业务系统快照，历史考试、训练、打卡记录都绑定本地 `users.id`，不能因为企微删除账号就删除本地用户。
- 企业微信控制“这个人当前是否还在组织内、是否可登录、是否可接收推送”。
- 本地管理员仍可提前创建新人账号，用于试岗、培训、考试、读书打卡。
- 新人正式入职后，如果企业微信新增了同手机号账号，需要自动或半自动绑定到本地已有账号，而不是创建第二个业务用户。
- 绑定后，后续推送优先使用 `wecom_userid`，登录匹配仍可用手机号兜底。

### 4.2 主身份与绑定关系

推荐身份关系：

- 本地主键：`users.id`，业务记录永远绑定它。
- 本地登录名：`users.username`，当前约定为手机号。
- 企业微信身份：`wecom_userid`，用于企微登录、消息推送、同步状态。
- 绑定规则：优先 `wecom_userid`，其次手机号 `mobile == users.username`。

绑定优先级：

1. 企微 `userid` 已绑定本地 `users.wecom_userid`：直接更新该用户快照。
2. 企微 `userid` 未绑定，但企微 `mobile` 能匹配本地 `users.username`：绑定到该本地用户，并写入 `wecom_userid`。
3. 企微 `mobile` 匹配不到本地用户：默认进入“企微新增待处理”，不自动创建，除非开启自动创建策略。
4. 企微 `mobile` 匹配到多个本地用户：标记冲突，禁止自动绑定，需要管理员处理。
5. 本地用户有 `wecom_userid`，但企微通讯录已找不到该 `userid`：按企微删除/离职处理。

新人试岗场景：

- 管理员可先在本地创建账号，`username` 填手机号，`employment_status` 可为 `试岗` 或 `试用`。
- 新人还没进企业微信时，`wecom_userid` 为空，仍可用本地账号密码登录、参加培训或考试。
- 新人正式入职后，企微新增同手机号成员。
- 下一次同步时，系统用手机号匹配到本地账号，写入 `wecom_userid`，并更新姓名、部门、岗位等企微信息。
- 绑定后，该用户可通过企微免登录，也可接收企业微信推送。

### 4.3 字段映射

建议从企业微信通讯录同步到本地快照：

| 本地字段 | 企业微信来源 | 说明 |
| --- | --- | --- |
| `username` | `mobile` | 当前本地用户名就是手机号 |
| `real_name` | `name` | 员工真实姓名 |
| `display_name` | `name` | 可与真实姓名一致 |
| `department` | 部门列表/部门名称 | 需要用部门 ID 再映射名称 |
| `position` | `position` | 岗位 |
| `status` | 成员状态 | 企业微信删除/禁用后映射为 `inactive` |
| `disabled` | 成员是否可用 | 企业微信删除/禁用后置为 `true`，禁止登录和推送 |
| `employment_status` | 自定义字段或本地规则 | 企微没有该字段时，本地维护；企微删除时统一改为 `离职` |
| `wecom_userid` | `userid` | 推荐新增，便于消息推送 |

字段覆盖策略：

- `real_name`、`display_name`、`department`、`position`：绑定企微后默认由企微覆盖。
- `username`：原则上不自动改，除非管理员确认手机号变更；因为它现在也是登录名。
- `role`：不由企微覆盖，仍由本地管理员控制。
- `is_newcomer`：不由企微直接覆盖，可由本地业务规则或管理员控制。
- `employment_status`：默认本地维护；当企微账号不存在或被删除时，强制改为 `离职`。
- `status` / `disabled`：企微存在且可用时可恢复为 `active` / `false`，企微删除或禁用时改为 `inactive` / `true`。

### 4.4 同步模式

推荐分三层：

- 首次初始化绑定：第一次接入企业微信时使用，只做匹配、绑定、补字段、列冲突，不直接清洗掉现有本地用户。
- 手动同步：管理员点击“从企业微信同步用户”，适合早期上线。
- 定时同步：每天凌晨同步用户快照，适合稳定运行。
- 登录时补全：用户企微登录成功后，顺手补齐 `wecom_userid`、姓名、部门、岗位等字段。

### 4.5 首次初始化绑定

当前系统已经通过 Excel/后台手工导入了大量本地用户。正式对接企业微信时，不能把企微通讯录重新全量拉下来覆盖本地用户，更不能重建用户，因为考试、训练、读书打卡、评分记录都绑定本地 `users.id`。

首次接入应使用“初始化绑定”流程：

1. 保留所有现有本地用户和 `users.id`。
2. 从企业微信拉取通讯录成员。
3. 使用企微手机号 `mobile` 匹配本地 `users.username`。
4. 匹配成功：给本地用户补写 `wecom_userid`，并按策略更新姓名、部门、岗位等快照字段。
5. 本地有、企微没有：标记为“本地未绑定企微”，首次初始化阶段不自动改离职、不自动禁用。
6. 企微有、本地没有：进入“企微新增待开通”，由管理员确认是否创建本地用户。
7. 手机号为空、手机号重复、同手机号不同 `userid`：进入冲突列表，人工处理。
8. 管理员确认预览结果后，再执行绑定和字段更新。

首次初始化绑定的目标：

- 把“已有本地账号”和“企微账号”对齐。
- 尽量复用本地已有用户，不制造重复账号。
- 不破坏历史业务数据。
- 把无法自动判断的情况留给管理员确认。

首次初始化完成后，才进入日常同步规则：

- 已绑定用户如果企微删除/禁用，本地可改为离职/禁用。
- 新增企微成员按手机号绑定本地预建账号，或进入待开通。
- 部门、岗位、姓名等字段按覆盖策略持续更新。

建议手动同步先做“预览再执行”：

- 预览本次会新增、绑定、更新、离职、禁用、冲突多少人。
- 管理员确认后再执行写库。
- 冲突项不自动写库，只展示原因和建议处理动作。

### 4.6 企微删除、禁用、离职处理

当本地用户已经绑定 `wecom_userid`，但同步时企业微信通讯录中找不到该成员，或者成员状态为禁用/离职：

- 不删除本地 `users` 记录。
- 将 `users.employment_status` 改为 `离职`。
- 将 `users.status` 改为 `inactive`。
- 将 `users.disabled` 改为 `true`。
- 保留 `wecom_userid` 和 `wecom_raw_json`，方便审计和排查。
- 后续不再给该用户发送企业微信学习/考试通知。
- 该用户历史考试、训练、打卡、评分记录保持可查。

如果企微后续重新添加同手机号账号：

- 如果 `userid` 和历史一致，直接恢复绑定并按企微状态更新。
- 如果 `userid` 变化但手机号相同，进入“疑似重新入职/账号重建”状态。
- 管理员确认后，可以把新 `userid` 绑定到原本地用户，并将 `disabled` 恢复为 `false`，`status` 恢复为 `active`。
- 是否把 `employment_status` 从 `离职` 改为 `试用`、`转正` 等，应按本地业务规则或管理员确认，不建议无脑自动改为转正。

### 4.7 本地提前创建账号后的企微绑定

这是本项目必须支持的核心场景：

1. 新人还没正式入职，管理员先在本地创建账号。
2. 新人用本地账号参加试岗培训、考试、读书打卡。
3. 后续企微通讯录新增该员工，手机号与本地 `username` 一致。
4. 同步时系统识别为“本地预建账号待绑定”，而不是“企微新增用户”。
5. 系统写入 `wecom_userid`，补齐姓名、部门、岗位。
6. 后续企微登录、企微推送都走这个本地用户。

建议管理端同步预览增加分类：

- `已绑定更新`：已有 `wecom_userid`，正常更新字段。
- `预建账号绑定`：企微手机号匹配本地未绑定账号，准备写入 `wecom_userid`。
- `企微新增待开通`：企微有，本地没有。
- `企微缺失转离职`：本地已绑定，但企微已不存在。
- `手机号冲突`：多个本地用户或多个企微用户手机号冲突。
- `缺少手机号`：企微用户没有手机号，无法匹配。

### 4.8 冲突处理

必须定义：

- 企微手机号为空：跳过，不创建本地用户。
- 本地手机号不存在：默认进入“企微新增待开通”，不自动创建，除非管理员开启自动创建。
- 本地存在多个相同手机号：不允许登录，并要求管理员清理数据。
- 企微存在多个相同手机号：不自动绑定，提示企微通讯录需要清理。
- 本地用户已绑定 A 企微 `userid`，但同手机号出现 B 企微 `userid`：标记疑似重新入职或企微账号重建，需要管理员确认。
- 用户离职/停用：将本地 `employment_status = 离职`、`disabled = true`，但历史考试、训练、打卡记录保留。
- 管理员账号：如果 `admin/super_admin` 不是手机号，继续允许密码登录；如果也是手机号，可同时支持企微登录。

### 4.9 同步审计和回滚

建议每次同步生成同步批次记录：

- 同步开始时间、结束时间。
- 拉取企微用户数量。
- 本地匹配数量。
- 新绑定数量。
- 更新数量。
- 转离职数量。
- 冲突数量。
- 执行人或定时任务标识。
- 原始结果摘要。

建议保存单人同步日志：

- 同步批次 ID。
- 本地 `user_id`。
- 企微 `userid`。
- 匹配方式：`wecom_userid` / `mobile` / `manual`。
- 动作：`update` / `bind` / `mark_left` / `disable` / `conflict` / `skip`。
- 变更前后字段快照。
- 错误原因。

这样后续如果企微通讯录误删，系统可以查清是哪一次同步把用户置为了离职，并支持管理员手动恢复。

## 5. 消息通知接入范围

### 5.1 用户端通知

优先级从高到低：

1. 试卷派发通知：管理员派发试卷后，通知用户去 `/papers/:assignmentId/take` 或 `/papers`。
2. 考试临近截止提醒：截止前 N 小时提醒未完成用户。
3. 考试提交成功通知：用户提交后提示“已提交，AI/系统正在评分”。
4. 考试评分完成通知：`paper_submissions.status` 变为 `graded` 后通知用户查看结果页。
5. 场景通关派发通知：如果继续使用 `exams`，创建通关任务后通知用户。
6. 场景通关结果通知：通关复盘或复核完成后通知用户查看结果。
7. 视频学习提醒：课程发布或指定用户未学习时提醒。
8. 读书/音频打卡提醒：每日打卡提醒、未完成提醒、补卡提醒。

### 5.2 管理端通知

建议补齐：

- 用户提交试卷后，如进入 `submitted` 待复核，通知管理员。
- AI 评分失败或长时间未完成，通知管理员处理。
- 有大量用户临近截止未完成，通知管理员。
- 用户同步失败、手机号冲突、企微接口失败，通知超级管理员。

### 5.3 消息落地建议

建议新增统一通知层，而不是每个业务模块直接拼企业微信 API：

- `backend/app/wecom_client.py`：封装 access_token、API 请求、错误处理。
- `backend/app/wecom_auth.py`：封装 OAuth 登录、userid/mobile 获取。
- `backend/app/wecom_push.py`：保留现有文件，替换 stub，实现应用消息发送。
- `backend/app/notification_service.py`：统一业务通知入口，例如 `notify_paper_assigned`、`notify_submission_graded`。

## 8. 数据库建议改造

### 8.1 用户表建议字段

推荐：

```sql
ALTER TABLE users
  ADD COLUMN wecom_userid VARCHAR(128) NULL COMMENT '企业微信成员UserID',
  ADD COLUMN wecom_synced_at DATETIME NULL COMMENT '最近一次企业微信同步时间',
  ADD COLUMN wecom_raw_json LONGTEXT NULL COMMENT '企业微信原始用户快照';
```

建议索引：

```sql
CREATE UNIQUE INDEX uk_users_wecom_userid ON users (wecom_userid);
```

注意：

- 如果历史数据中存在空字符串，应统一使用 `NULL`，避免唯一索引冲突。
- 如果暂时坚持不落 `wecom_userid`，消息推送时每次都要通过手机号查企微 userid，稳定性和性能会差一些。

### 8.2 通知日志表建议

建议新增通知日志表：

```sql
CREATE TABLE notification_logs (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  channel VARCHAR(32) NOT NULL,
  event_type VARCHAR(64) NOT NULL,
  recipient_user_id BIGINT NULL,
  recipient_wecom_userid VARCHAR(128) NULL,
  business_type VARCHAR(64) NOT NULL,
  business_id BIGINT NULL,
  status VARCHAR(16) NOT NULL DEFAULT 'pending',
  payload_json LONGTEXT NULL,
  response_json LONGTEXT NULL,
  error TEXT NULL,
  sent_at DATETIME NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  KEY idx_notification_logs_recipient (recipient_user_id, created_at),
  KEY idx_notification_logs_business (business_type, business_id),
  KEY idx_notification_logs_status (status, created_at)
);
```

作用：

- 追踪哪些消息发过、发给谁、成功或失败。
- 支持失败重试。
- 管理端可排查“为什么用户没收到通知”。

## 9. 企业微信配置项

后续 `.env` 建议增加：

```env
WECOM_ENABLED=false
WECOM_CORP_ID=
WECOM_AGENT_ID=
WECOM_APP_SECRET=
WECOM_CONTACT_SECRET=
WECOM_REDIRECT_URI=
WECOM_FRONTEND_CALLBACK_URL=
WECOM_TOKEN_CACHE_SECONDS=7000
WECOM_REQUEST_TIMEOUT_SECONDS=10
WECOM_AUTO_CREATE_USER=false
WECOM_SYNC_DISABLED_USERS=true
```

说明：

- `WECOM_APP_SECRET`：自建应用 secret，用于应用消息和应用授权场景。
- `WECOM_CONTACT_SECRET`：通讯录 secret，用于同步成员、部门等通讯录信息。
- `WECOM_REDIRECT_URI`：企业微信后台配置的可信回调地址。
- `WECOM_FRONTEND_CALLBACK_URL`：后端登录成功后跳回前端的地址。

## 10. 多系统统一登录建议

如果后续有很多系统都要接企业微信，建议不要每个系统都完整实现一遍企业微信 OAuth。

推荐分两期：

第一期：

- 每个系统各自接企业微信登录。
- 统一身份规则：企微手机号匹配本地用户名。
- 统一配置命名、统一错误提示、统一回调模式。

第二期：

- 建一个轻量 SSO 服务。
- 所有系统跳转到统一 SSO。
- SSO 负责企业微信 OAuth、手机号/用户映射、登录态。
- 各业务系统只信任 SSO 返回的一次性 ticket 或内部 token。

这样后续多个系统不会各自维护一套企微登录逻辑。

## 11. 验收清单

### 11.1 登录验收

- 企业微信内打开应用，未登录用户可自动完成登录。
- 已登录用户刷新页面不需要重新登录。
- 企微手机号能匹配 `users.username`。
- 匹配不到本地用户时，有明确提示。
- 本地禁用用户无法通过企微登录。
- 管理员密码登录仍可用。

### 11.2 用户同步验收

- 能预览企微用户同步结果。
- 能识别新增、更新、停用、手机号缺失、手机号冲突。
- 同步后部门、岗位、姓名、在职状态策略符合预期。
- 离职或禁用不删除历史记录。

### 11.3 消息通知验收

- 试卷派发后用户能收到企业微信消息。
- 消息中的链接能直接打开对应页面。
- 用户提交后管理员能收到待复核提醒。
- 评分完成后用户能收到结果提醒。
- 推送失败能看到失败原因并支持重试。
- 重复推送有控制，不会短时间刷屏。

### 11.4 回归验收

- 原账号密码登录不受影响。
- 原 JWT 鉴权不受影响。
- 用户端 `/home`、`/papers`、`/magic-academy`、训练记录等页面仍可正常访问。
- 管理端用户管理、试卷派发、待复核、魔学院管理仍可正常访问。
