# 企业微信接入开发文档

> 企业微信登录、应用消息推送、通讯录同步在当前版本均已落地。本文档记录现状、配置项、关键代码、运维细节，便于交接与排错。

## 1. 总体能力

- 企业微信扫码 / 工作台免密登录，复用系统 JWT。
- 通讯录同步走第三方 HR 系统：管理端"预览 → 执行"按需触发；HR 返回的数据已带 `wecom_userid`，本地不再需要单独拉企微通讯录定时同步。
- 试卷派发 / 提交 / 评分 / 截止、通关派发 / 截止、魔学院读物与视频派发等多场景应用消息推送。
- 推送监控页：状态 / 类型筛选、详情、统计、批量删除。

## 2. 配置项

`backend/.env` 现行变量（默认值见 [backend/app/config.py](backend/app/config.py)）：

```env
# --- 总开关 ---
WECOM_ENABLED=false
WECOM_PUSH_ENABLED=true
WECOM_LOGIN_ENABLED=true
WECOM_SYNC_ENABLED=false          # 仅用于直接调用企微通讯录调试（fetch_all_members）

# --- 应用 ---
WECOM_CORP_ID=
WECOM_AGENT_ID=
WECOM_APP_SECRET=
WECOM_CONTACT_SECRET=             # 需要直连通讯录调试时才填

# --- 登录回调 ---
WECOM_REDIRECT_URI=               # 企业微信后台配置的可信回调 URL
WECOM_FRONTEND_CALLBACK_URL=      # 后端登录成功后跳回前端的页面（默认 /auth/wecom/callback）
WECOM_FRONTEND_BASE_URL=          # 前端站点根地址，用于消息正文里的链接

# --- 行为细节 ---
WECOM_ADMIN_USERIDS=              # 逗号分隔，用于"通知超级管理员"
WECOM_TOKEN_CACHE_SECONDS=7000
WECOM_REQUEST_TIMEOUT_SECONDS=10
WECOM_SYNC_DISABLED_USERS=true    # HR 同步时是否处理被禁用账号
WECOM_AUTO_REDIRECT_IN_CLIENT=true # 在企微 H5 内打开 /login 自动跳转 OAuth
WECOM_STATE_TTL_SECONDS=600
WECOM_SYNC_PROTECTED_STATUSES=试岗,离职  # 这些 employment_status 不允许被同步覆盖

# --- 第三方 HR 同步 ---
EMPLOYEE_SYNC_ENABLED=true
EMPLOYEE_SYNC_BASE_URL=
EMPLOYEE_SYNC_APP_ID=
EMPLOYEE_SYNC_PARTNER_PRIVATE_KEY=
EMPLOYEE_SYNC_HR_PUBLIC_KEY=
EMPLOYEE_SYNC_TIMEOUT_SECONDS=15
EMPLOYEE_SYNC_AUTO_CREATE_USER=true
```

`Settings.wecom_login_ready` / `wecom_push_ready` / `wecom_sync_ready` / `employee_sync_ready` 等派生属性会综合判断必填字段是否齐全；只有 ready 才会真正注册路由 / 调用接口。

## 3. 登录流程

代码：[backend/app/auth.py](backend/app/auth.py) + [backend/app/wecom_auth.py](backend/app/wecom_auth.py)

1. 用户在企业微信工作台进入应用，前端发现没有本系统 token。
2. 前端 `LoginPage` 通过 `/api/auth/providers` 拿到 `wecom_enabled / wecom_auto_redirect_in_client`。
3. `wecom_auto_redirect_in_client = true` 时直接 `location.replace('/api/auth/wecom/start?redirect=/home')`。
4. 后端 `wecom_start` 校验 `wecom_login_ready`，组合 `state` 跳企微授权页。
5. 企微回调 `/api/auth/wecom/callback?code=...&state=...`。
6. 后端 `login_by_code(db, code)`：
   - 调用企微 `/auth/getuserinfo` + `/user/get` 拿 `userid` / `mobile`。
   - 优先按 `users.wecom_userid` 匹配；其次按手机号 `users.username`；命中后写入 / 校准 `wecom_userid`。
   - 匹配不到时返回带 `WecomAuthError` 的 302，前端在 `/login` 显示中文错误。
7. 命中后 `create_token` 签发 JWT，302 至 `WECOM_FRONTEND_CALLBACK_URL?token=...&expires_at=...&user_json=...`。
8. 前端 `WecomCallbackPage` 把字段写进 localStorage，再跳到 `state` 中保存的目标页。

## 4. 用户表与同步

### 4.1 当前 `users` 字段

模型在 [backend/app/models.py](backend/app/models.py)：

| 字段 | 说明 |
| --- | --- |
| `id` | 业务主键 |
| `username` | 登录名，约定为手机号 |
| `password_md5` | 密码 |
| `display_name` / `real_name` | 昵称 / 真实姓名 |
| `department` / `position` | 部门 / 岗位 |
| `role` | `user` / `admin` / `super_admin` |
| `is_newcomer` | 新人标记 |
| `employment_status` | 在职状态（试岗 / 试用 / 在职 / 离职 等） |
| `status` | `active` / `inactive` |
| `disabled` | 是否禁止登录 |
| `wecom_userid` | 企微用户唯一 ID（带 unique 索引） |
| `wecom_synced_at` | 最近一次同步时间 |
| `wecom_raw_json` | 企微原始快照（审计用） |

### 4.2 同步入口

代码：[backend/app/employee_sync.py](backend/app/employee_sync.py)、[backend/app/employee_open_client.py](backend/app/employee_open_client.py)、[backend/app/users_api.py](backend/app/users_api.py)

- `POST /api/admin/users/employee-sync/preview`：拉取第三方 HR 通讯录，按手机号 / 姓名 / `wecom_userid` 多路匹配，返回分类预览。
- `POST /api/admin/users/employee-sync/execute`：携带 `preview_token` 真正写库；批次记录到 `wecom_sync_batches`，单人记录到 `wecom_sync_entries`。
- `GET /api/admin/users/employee-sync/search?keyword=...`：按关键词查 HR 员工。
- HR 同步是 **按需触发**，没有定时任务。HR 接口返回的数据自带 `wecom_userid`，因此本地不再需要直接拉企微通讯录跑日常同步（如需调试可调 `WecomClient.fetch_all_members`）。

### 4.3 分类与动作

预览结果按下面分桶（与 `wecom_sync_entries.action` 对齐）：

| 分类 | 触发条件 | 写库动作 |
| --- | --- | --- |
| 已绑定更新 | `wecom_userid` 已存在 | `update` |
| 预建账号绑定 | 手机号匹配本地未绑定账号 | `bind` + 字段更新 |
| 同名异号绑定 | 同姓名、不同手机号、唯一匹配 | `bind`（受保护字段不覆盖） |
| HR 新增待开通 | HR 有，本地无；`EMPLOYEE_SYNC_AUTO_CREATE_USER=true` 时自动建账号 | `create` 或 `skip` |
| HR 缺失转离职 | 本地已绑定但 HR 找不到 | `mark_left` + `disable`（保留 `wecom_userid` 与 `wecom_raw_json`） |
| 手机号冲突 | 多个本地或多个 HR 同手机号 | `conflict`，不写库 |
| 缺少手机号 | HR 用户没有手机号 | `skip` |

`WECOM_SYNC_PROTECTED_STATUSES`（默认 `试岗,离职`）覆盖的用户不会被自动改 `employment_status`，避免误把 HR 仍在试岗的人切回正式状态。

> **表命名说明**：`wecom_sync_batches` / `wecom_sync_entries` 名字带 `wecom` 是历史命名，当前实际由 HR 同步在写，未做改名（避免数据迁移成本）。代码里类同名 `WecomSyncBatch` / `WecomSyncEntry`。

## 5. 推送链路

### 5.1 服务分层

- [wecom_client.py](backend/app/wecom_client.py)：access_token 缓存、应用消息发送、错误码映射；进程内单例 + 共享 httpx。
- [wecom_push.py](backend/app/wecom_push.py) / [wecom_push_bulk.py](backend/app/wecom_push_bulk.py)：把业务对象拼成应用消息 payload；批量推送编排。
- [notification_service.py](backend/app/notification_service.py)：业务通知统一入口（试卷派发 / 截止 / 提交、通关派发 / 截止、管理员告警等），先写 `notification_logs`（status=pending）→ 调推送 → 再用 `_finalize_log` 标记 `sent / failed` 并写 `response_json` / `error`。同文件还提供 `resend_notification` / `resend_notifications_bulk` 给推送监控页统一重推。
- [magic_push_service.py](backend/app/magic_push_service.py) + [magic_auto_actions.py](backend/app/magic_auto_actions.py)：魔学院推送编排与自动派发。

### 5.2 已接入的事件类型

`notifications_api.EVENT_TYPE_META`：

| event_type | 描述 |
| --- | --- |
| `paper_assigned` | 试卷派发通知 |
| `paper_deadline_reminder` | 试卷截止提醒 |
| `paper_submission_received` | 试卷提交进度 |
| `exam_assigned` | AI 通关派发 |
| `exam_deadline_reminder` | AI 通关截止提醒 |
| `magic_video_assigned` | 魔学院课程派发 |
| `magic_reading_published` | 魔学院读物派发 |

未在 `EVENT_TYPE_META` 登记的事件也会出现在筛选下拉里（`label = event_type`），方便排查临时事件。

### 5.3 推送监控（含统一重推）

代码：[backend/app/notifications_api.py](backend/app/notifications_api.py)、前端 [NotificationsTab.jsx](frontend/src/components/admin/NotificationsTab.jsx)

- 列表 / 详情 / 统计 / 事件类型 / 批量删除（`POST /api/admin/notifications/bulk-delete`）。
- **单条重推**：`POST /api/admin/notifications/{log_id}/resend` —— 监控页失败行的「重推」按钮触发；原日志保留作历史记录，重推过程会写一条新的 `notification_logs`。
- **批量重推**：`POST /api/admin/notifications/bulk-resend` —— 监控页多选后点「批量重推」；`status=sent` 的会被自动跳过，其余按 `event_type` 路由到对应业务的 `notify_*`。
- 重推按 `event_type` 分发：试卷派发 / 试卷截止提醒 / 试卷提交、AI 通关派发 / 截止、魔学院课程 / 读物。业务对象（试卷派发、通关、视频、读物）已删除时返回 `skipped` 不写新日志。
- 失败时优先排查：60020（IP 白名单）、`recipient_wecom_userid` 是否为空、`access_token` 是否过期、`agent_id` 是否正确。

此外，AI 通关派发面板（[ExamAssignmentsPanel.jsx](frontend/src/components/admin/ExamAssignmentsPanel.jsx)）也提供单条「推送」按钮，调用 `POST /api/admin/exams/{exam_id}/wecom-push` 复用同一条批量推送链路，便于直接在派发页快速重推单条。

## 7. 多系统统一登录建议

当前实现是"单系统接企微"，后续如果要接入更多系统：

- 第一阶段：维持每系统接企微，但统一 `users.username = mobile`、统一回调 / 错误提示 / 配置命名。
- 第二阶段：抽出独立 SSO 服务，所有业务系统跳到 SSO，由 SSO 负责企微 OAuth + 手机号映射 + 颁发短期 ticket，业务系统只信 ticket。

## 8. 验收清单

### 8.1 登录

- [ ] 企微工作台未登录用户自动完成登录。
- [ ] 已登录用户刷新页面无需重新登录。
- [ ] 企微手机号 / `userid` 能匹配 `users`。
- [ ] 匹配不到时显示中文错误 `WecomAuthError.detail`。
- [ ] 本地禁用用户无法通过企微登录。
- [ ] 管理员密码登录仍可用。

### 8.2 同步

- [ ] 预览能识别已绑定更新 / 预建账号绑定 / HR 新增待开通 / HR 缺失转离职 / 手机号冲突 / 缺少手机号。
- [ ] 受保护 `employment_status` 不被自动覆盖。
- [ ] 同步批次和单人记录均落 `wecom_sync_batches` / `wecom_sync_entries`。
- [ ] 没有任何定时同步进程在跑（`wecom_daily_sync_worker` 已下线）。

### 8.3 推送

- [ ] 试卷派发 / 截止 / 提交 / 评分完成、通关派发 / 截止、魔学院读物 / 视频均能收到消息。
- [ ] 消息中的链接可在 PC 浏览器与企微 H5 内打开。
- [ ] 失败原因在推送监控可见，可在监控页单条 / 批量「重推」。
- [ ] 推送监控批量删除工作正常，仅清日志、不影响业务。
- [ ] AI 通关派发面板的"推送"按钮可对单条派发主动触发推送，结果在监控页可查。

### 8.4 回归

- [ ] 账号密码登录 / JWT 鉴权 / 用户端各页面 / 管理端各 Tab 不受影响。
- [ ] 关闭 `WECOM_ENABLED` 后，登录 / 推送 / 同步路径自动回退到禁用态而不是报错。
