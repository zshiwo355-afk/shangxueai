# 复盘输出规则

RULE_ID: RULE_REVIEW_OUTPUT  
DOC_TYPE: rule  
VERSION: 2026-06-11  
PRIORITY: 95  
适用阶段: 最终复盘  
关键词: 复盘, 成交, 意向客户, 未成交, 优点, 不足, 痛点, 建议

## 一、复盘目标

最终复盘要让新人知道：
1. 最终为什么成交、形成意向或未成交；
2. 客户真实痛点是什么；
3. 自己做得好的地方；
4. 哪些环节丢分；
5. 下次训练重点是什么；
6. 如果是意向客户，下一步应该如何跟进。

## 二、必须输出字段

最终复盘 JSON 必须包含：

```json
{
  "result": "成交/意向客户/未成交",
  "score": 0,
  "is_pass": true,
  "dimension_scores": {
    "need_probe": 0,
    "brand_value": 0,
    "product_recommendation": 0,
    "objection_handling": 0,
    "emotion_control": 0,
    "closing": 0,
    "compliance": 0
  },
  "customer_pain_points": [],
  "strengths": [],
  "weaknesses": [],
  "key_turning_points": [],
  "deal_reason": "",
  "lost_reason": "",
  "compliance_risks": [],
  "next_training_focus": [],
  "suggested_better_replies": []
}
```

## 三、输出要求

- 主结果只能是“成交”“意向客户”或“未成交”。
- 不要输出第三种以外的结果，例如“待定”“部分成交”。
- score 必须是 0-100 的整数。
- dimension_scores 各项相加应尽量等于 score。
- 痛点要从客户话语和隐藏画像推断，不要胡编。
- strengths 至少 2 条。
- weaknesses 至少 2 条。
- suggested_better_replies 最多 3 条，不能过度替新人写整场话术。
- 出现合规风险必须写入 compliance_risks。
- 不得因为未满固定轮次而自动判定未成交。

## 四、成交复盘重点

如果成交：
- 说明成交关键动作；
- 说明客户被解决的核心痛点；
- 说明哪个阶段推动了信任上升；
- 说明仍可优化的地方。

## 五、意向客户复盘重点

如果是意向客户：
- 说明客户为什么产生意向；
- 说明客户仍未成交的关键顾虑；
- 说明新人已经完成了哪些有效推进；
- 给出下一步跟进建议，例如发资料、发方案、约回访、确认决策人、确认预算或数量。

## 六、未成交复盘重点

如果未成交：
- 说明关键丢单点；
- 区分是需求没问清、价值没讲清、异议没处理、没有推进动作，还是合规问题；
- 给出下一次训练重点；
- 如果对话较短，可以提示“本次对话较短，建议继续训练完整链路”，但不要把轮次少作为唯一失败原因。

## 七、语言风格

复盘语言要直接、具体、可执行。避免空话，例如“需要加强沟通能力”。应写成“下一次要先问清客户是送礼还是自饮，再决定推荐产品”。
